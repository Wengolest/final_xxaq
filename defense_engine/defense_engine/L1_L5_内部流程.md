# L1→L5 内部执行流程

五层检测器遵循统一模式：**程序化检查 → 规则引擎匹配 → 汇总计算**，通过 `scoring.compute_layer_result()` 将各检查结果转为统一的 `LayerCheckResult`。

---

## 通用执行模板

```
evaluate(ctx, engine, **kwargs)
  │
  ├── 1. 运行 N 个 _check_*() 方法，产出 CheckFlag 列表
  │      每个 check 返回 CheckFlag(check_type, severity, description)
  │      severity ∈ {block, warn, log, pass}
  │
  ├── 2. 调用 rule_engine.evaluate(ctx, layer_prefix="XX")
  │      规则引擎从上下文字段匹配条件，产出额外 CheckFlag(rules)
  │
  ├── 3. _summarize(flags, trust_in, t_start)
  │      └── compute_layer_result(layer, flags, trust_in, t_start)
  │           ├── 按 source 分类加权: programmatic(0.25/0.08) / rule(0.30/0.10) / semantic(0.40/0.15)
  │           ├── 判定 action: 有 block → block, 有 warn → warn, 否则 pass
  │           ├── trust_new = max(0.0, trust_in − risk)
  │           └── 返回 LayerCheckResult(passed, action, risk_score, trust_level, ...)
  │
  └── 4. 返回 LayerCheckResult
```

## L1 源头治理 — `layer1_source_governance.py:evaluate()`

**7 项程序化检查，顺序执行：**

| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| 1 | `_check_source()` | 来源白名单/黑名单（anonymous→warn, dark_web→block） | block/warn |
| 2 | `_check_zero_width()` | 零宽字符 U+200B/U+200C/U+FEFF/U+202E 等 | block |
| 3 | `_check_pseudo_system()` | 伪系统指令 `<|system|>`, `### SYSTEM`, `[system]` 等 | block |
| 4 | `_check_encoding_obfuscation()` | `\xHH` hex转义、`%HH` URL编码、base64混淆 | block/warn |
| 5 | `_check_rtl_attack()` | RTL覆盖字符 U+202E 改变文本渲染顺序 | block |
| 6 | `_check_repetition()` | 同内容在短窗口内重复>3次（防暴力试探） | warn |
| 7 | `_check_file_size()` | 仅当内容>1MB触发（防上下文窗口溢出） | warn |

**之后**: `engine.evaluate(ctx, layer_prefix="SG")` 匹配 SG 前缀规则。

**层权重**: 使用标准 `DEFAULT_WEIGHTS`（block=0.25, warn=0.08, rule_block=0.30, rule_warn=0.10）。

**设计动机**: L1 是"入口安检"——在内容进入系统前拦截格式层和来源层攻击。这些攻击（零宽字符、编码混淆）在语义层面无法检测，必须在原始字节/字符层面处理。

---

## L2 模型交互 — `layer2_model_interaction.py:evaluate()`

**6 项程序化检查：**

| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| 1 | `_check_dangerous_separators()` | `ignore all`, `[/INST]`, `<<SYS>>`, `-----` 等 | block |
| 2 | `_check_instruction_confusion()` | 输入中混入标记语法混淆模型（仅 data 类内容触发） | warn |
| 3 | `_check_sensitive_behavior()` | 危险行为关键词：bypass/override/disable security 等 | block |
| 4 | `_check_context_escalation()` | 上下文越权试探：system prompt/reveal rules/internal config | block/warn |
| 5 | `_check_pii()` | 身份证/手机号/银行卡/API key/AWS key 正则 | block |
| 6 | `_check_context_length()` | 输入>10000字符警告，>32000拦截 | warn/block |

**之后**: `engine.evaluate(ctx, layer_prefix="MI")` 匹配 MI 前缀规则。

**层权重**: 使用标准 `DEFAULT_WEIGHTS`。

**设计动机**: L2 处理"文本层面"的攻击——即使用户输入格式正确，内容可能包含指令混淆、越权试探或敏感信息泄露。L1 处理字符，L2 处理语义。

---

## L3 记忆控制 — `layer3_memory_control.py:evaluate()`

**根据操作类型分支，再统一做内容危险检测：**

### 写入路径（4 项）：
| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| 1 | `_check_write_risk()` | 来源信任度<0.3→block，0.3-0.5→warn | block/warn |
| 2 | `_check_source_annotation()` | 自动标注来源信任级别到 entry 元数据 | log |
| 3 | `_assign_ttl()` | 分配倒计时生存时间（低信任源默认24h） | log |
| 4 | `_check_conflict()` | 新写入与已有条目语义冲突检测 | warn |

### 读取路径（2 项）：
| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| 1 | `_check_read_filter()` | 过滤 quarantined/archived/expired 条目 | block(filtered) |
| 2 | `_check_expiration()` | 标记过期条目为 archived 状态 | log |

### 统一检测：
| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| - | `_check_content_danger()` | 内容包含"跳过安全/自动批准/无需确认"等危险指令 | block |

**之后**: `engine.evaluate(ctx, layer_prefix="MC")`，ctx 包含 `written_at`, `ttl`, `entry_status`, `source_trust`, `source_trust_divergence` 等记忆特化字段。

**层权重**: 使用 `MEMORY_LAYER_WEIGHTS`（block=0.28, warn=0.08, rule_block=0.32, rule_warn=0.10）。

**设计动机**: L3 是唯一跨会话持久化的防御层——一旦恶意记忆写入，污染会持续影响后续所有会话。因此采用代理模式强制所有记忆操作经过安全检查。

---

## L4 工具约束 — `layer4_tool_constraint.py:evaluate()`

**仅当 `tool_call` 参数非空时执行 6 项检查：**

| # | 方法 | 检测内容 | 严重级别 |
|---|------|---------|---------|
| 1 | `_check_tool_whitelist()` | 工具名不在白名单→block | block |
| 2 | `_check_permission()` | 工具所需权限超过当前 trust_level 授权 | block |
| 3 | `_check_high_risk()` | 参数匹配危险模式：`rm -rf`, `/etc/passwd`, `| bash`, `> /dev/` 等 | block |
| 4 | `_check_params_danger()` | 参数包含路径遍历 `../`、命令注入 `;` `&&` 等 | block/warn |
| 5 | `_check_rate_limit()` | 同一工具30秒内调用超过阈值→block | block |
| 6 | `_check_audit()` | 高风险工具调用自动记录完整审计日志 | log |

**之后**: `engine.evaluate(ctx, layer_prefix="TC")`。

**层权重**: 使用 `TOOL_LAYER_WEIGHTS`（block=0.30, warn=0.10, rule_block=0.35, rule_warn=0.12）——工具误调用代价最大，权重高于其他层。

---

## L5 决策监督 — `layer5_decision_supervision.py:evaluate()`

**先收集前四层结果，再做全局决策：**

| 步骤 | 方法 | 检测内容 | 严重级别 |
|------|------|---------|---------|
| - | 收集阶段 | 统计前四层的 block 数、warn 数、累积风险 | — |
| 1 | `_check_circuit_breaker()` | 连续3+请求被block→熔断打开，拒绝所有请求 | block |
| 2 | `_check_cross_validation()` | 跨层交叉验证：多个层对同一攻击发出warn→升级为block | block/warn |
| 3 | `_check_audit()` | 累积风险>0.5时分叉→完整审计复核 | block/warn |
| 4 | `_check_anomaly()` | 检测异常行为模式：请求频率陡增、风险分斜率突变 | warn |
| 5 | `_check_final_arbitration()` | 当各层意见分歧时（有block有pass）做最终仲裁 | block/warn |
| - | `_update_state()` | 更新熔断器计数器、滑动窗口统计等持久状态 | log |

**之后**: `engine.evaluate(ctx, layer_prefix="DS")`，ctx 包含 `cumulative_risk`, `prior_blocks`, `consecutive_blocks`, `high_risk_ratio` 等全局状态字段。

**层权重**: 使用 `DECISION_LAYER_WEIGHTS`（semantic_block=0.40, semantic_warn=0.15 显式配置，为 AI 检测预留）。

**设计动机**: L5 是"安全委员会"角色——它不直接检测攻击内容，而是审查前四层的检测质量，通过跨层交叉验证降低单层误报/漏报，并在系统级异常时启动熔断保护。

---

## 层间数据流

```
trust=1.0 → L1(源头治理) → trust=1.0−r₁ → L2(模型交互) → trust=1.0−r₁−r₂
    │                                                         │
    │    ┌────────────────────────────────────────────────────┘
    │    ▼
    │   Agent 处理 ──→ 可能触发工具调用 → L4(工具约束) trust=...−r₄
    │    │
    │    └──→ 可能触发记忆操作 → L3(记忆控制) trust=...−r₃
    │
    ▼
  L5(决策监督) 收集 L1-L4 全部结果 → 交叉验证 → 熔断判断 → 最终输出
```

`trust_level` 从 1.0 开始逐层扣减，`risk_score` 从 0 开始逐层累加。两者是同一组 CheckFlag 加权计算的两种表示。L5 看到的是前四层的"全景图"——它用 `cumulative_risk` 和 `prior_blocks` 做最终仲裁。

# 已设计但未实现的大模型判断模块

以下列出了框架中理论上需要大模型语义判断、但当前仅为预留接口或占位实现的模块。

## 1. SemanticScorer 抽象基类 + PassThroughScorer 占位

**位置**: `scoring.py:197-273`

这是最核心的未实现模块。`SemanticScorer` 是一个完整的抽象基类，设计用于接入真实 ML 模型：

```python
class SemanticScorer(ABC):
    @abstractmethod
    def score(self, content: str, context: Optional[dict] = None) -> SemanticScore:
        ...
```

文档中明确列出未来可集成的方案：
- **Llama Guard 3 / Prompt Guard** — Meta 的专用安全分类器
- **自训练 ONNX/Transformers 分类模型** — 本地部署的轻量模型
- **远程安全审核 API** — 调用外部安全服务

**当前状态**: 唯一实现是 `PassThroughScorer`，它 **无条件返回 safe**（severity="log"），永不会触发任何拦截或警告。且该类 **从未被任何文件导入或调用**——整个 AI 检测链路完全断开。

## 4. 6 个攻击族 — 无任何检测逻辑

**位置**: `defense_types.py:50-63`

`AttackFamily` 枚举定义了 12 个攻击族，但以下 6 个**没有任何程序化检测或规则覆盖**：

| 攻击族 | 枚举值 | 说明 |
|--------|--------|------|
| 载荷拆分 | `PAYLOAD_SPLIT` | 将恶意指令拆分到多轮对话中 |
| Skill/MCP投毒 | `SKILL_MCP_POISONING` | 污染 Skill 定义或 MCP 工具描述 |
| 思维链攻击 | `CHAIN_OF_THOUGHT_ATTACK` | 利用模型推理过程注入恶意逻辑 |

| 供应链攻击 | `SUPPLY_CHAIN` | 污染Agent依赖的第三方工具/插件 |

## 总结：两条"真空带"

### 真空带 1 — 语义检测链

```
SemanticScorer 接口 (已设计)
    ↓ (断开: 无真实实现, PassThroughScorer 永不被调用)
semantic_block/warn 权重 (已设计, 0.40/0.15)
    ↓ (断开: 无 CheckFlag(source="semantic"))
compute_layer_result() semantic 分支 (死代码)
    ↓ (断开: 无规则使用 pattern_type="semantic")
rule_engine semantic 匹配 (硬编码 return False)
```

整条语义检测链的**架构已完整铺好**（接口→权重→评分分支→规则模式），但**没有任何一个节点被激活**。接入一个真实的 `SemanticScorer` 实现（如 Llama Guard）后，整条链路会自动贯通——这是框架设计上的刻意预留。

### 真空带 2 — 高级攻击族检测

6 个攻击族（PAYLOAD_SPLIT、SKILL_MCP_POISONING、CHAIN_OF_THOUGHT_ATTACK、OPINION_POISONING、MULTI_AGENT_POISONING、SUPPLY_CHAIN）完全无检测覆盖。这些攻击的共同特征是**不依赖危险关键词或格式异常**，而是利用多步交互、语义偏转或信任链传播——本质上需要大模型的上下文理解能力才能识别。

### 为什么当前可以不依赖大模型

当前框架在**无 ML 模型**的情况下仍能拦截大部分攻击，原因是：
- L1/L2/L4 的攻击向量（零宽字符、编码混淆、危险分隔符、高危命令模式）有**确定的字节/正则特征**，不需要语义理解
- L3/L5 通过信任度衰减和交叉验证提供**统计层面的防御**，不依赖内容理解
- 42 条内置规则覆盖了**有明确文本模式**的攻击场景

但上述两条真空带代表了框架的能力上限——要突破这个上限，必须接入真实语义检测模型。
